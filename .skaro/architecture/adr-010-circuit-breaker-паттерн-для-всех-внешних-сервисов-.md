# ADR-010: Circuit Breaker паттерн для всех внешних сервисов с автовосстановлением через 60 секунд

**Status:** accepted
**Date:** 2026-03-29

## Context
Приложение зависит от множества внешних сервисов: Gemini API, Google Drive, Dropbox, OneDrive, WebDAV-провайдеры, CloudKit, SMB. При деградации одного сервиса без защитного механизма приложение будет продолжать отправлять запросы, накапливая ошибки, блокируя UI и расходуя заряд батареи. Необходим механизм graceful degradation.

## Decision
Для всех внешних сервисов реализован Circuit Breaker: 3 последовательные ошибки переводят сервис в статус `.degraded`, пользователь уведомляется. Автоматическое восстановление через 60 секунд (переход в `.halfOpen`, пробный запрос). `CloudProviderHealthMonitor` управляет состоянием облачных провайдеров. `TTSService` при деградации Gemini TTS автоматически переключается на `AVSpeechSynthesizer`. Все ошибки типизированы через `ErrorCode` с кодом, описанием и recovery hint.

## Alternatives
1. **Простой retry с exponential backoff без circuit breaker:** Меньше сложности реализации — отклонён, так как при длительной недоступности сервиса приложение будет продолжать попытки, блокируя очередь запросов и расходуя ресурсы. Без circuit breaker невозможно быстро переключиться на fallback.
2. **Ручное управление состоянием сервисов пользователем:** Пользователь сам решает, когда повторить — отклонён, так как перекладывает техническую проблему на пользователя, ухудшает UX и не соответствует ожиданиям от современного мобильного приложения.
3. **Timeout без circuit breaker:** Каждый запрос имеет timeout, после которого показывается ошибка — отклонён, так как при системной деградации сервиса каждый запрос будет ждать полного timeout (10-30 секунд) перед показом ошибки, что неприемлемо для UX.

## Consequences
- Positive: Быстрый failover при деградации сервиса. Снижение нагрузки на деградирующий сервис. Автоматическое восстановление без участия пользователя. Чёткая типизация ошибок через `ErrorCode` упрощает диагностику.
- Negative: Дополнительная сложность реализации `CloudProviderHealthMonitor`. Порог в 3 ошибки и таймаут 60 секунд выбраны эмпирически и могут не оптимально подходить для всех сервисов (CloudKit vs Gemini имеют разные характеристики надёжности).
- Risks: Ложные срабатывания circuit breaker при кратковременных сетевых сбоях могут привести к деградации сервиса, который уже восстановился. Необходима тщательная настройка порогов для каждого типа сервиса. Состояние circuit breaker не персистируется — при перезапуске приложения счётчик ошибок сбрасывается.

# ADR-008: Запрет хранения coverData в SwiftData — только coverPath с файлом на диске

**Status:** accepted
**Date:** 2026-03-29

## Context
Библиотека может содержать тысячи книг. Каждая обложка занимает от 50KB до 500KB. Хранение `coverData: Data` непосредственно в SwiftData модели `Book` приведёт к загрузке всех обложек в память при выполнении `@Query`, что нарушит performance contract P95 < 300ms для 1000 книг.

## Decision
Поле `coverData: Data` в модели `Book` явно запрещено на уровне архитектурных инвариантов. Обложки хранятся исключительно в `Documents/Covers/{bookID}.jpg`. В SwiftData хранится только `coverPath: String`. Загрузка обложек в `LibraryView` и `BookCardView` осуществляется через `AsyncImage` с lazy loading. Проверяется через `check_refs.py`.

## Alternatives
1. **Хранение coverData в SwiftData как External Storage (`@Attribute(.externalStorage)`):** SwiftData автоматически выносит большие Data на диск — отклонён, так как SwiftData External Storage имеет ограниченную поддержку в iOS 17 и непредсказуемое поведение при миграции схемы. Явное управление файлами надёжнее.
2. **Хранение обложек в отдельной SwiftData сущности CoverRecord:** Нормализация данных — отклонён, так как создаёт дополнительный join при каждом запросе списка книг, усложняет модель без существенных преимуществ перед файловым хранением.
3. **NSCache для обложек в памяти с загрузкой из SwiftData:** Кэширование в памяти — отклонён, так как не решает проблему начальной загрузки: при первом `@Query` все `coverData` всё равно загружаются из SQLite в память.

## Consequences
- Positive: Выполнение performance contract P95 < 300ms для 1000 книг. Снижение размера SwiftData базы данных. `AsyncImage` обеспечивает автоматическое lazy loading и кэширование на уровне системы.
- Negative: Необходимо управлять жизненным циклом файлов обложек вручную (создание при импорте, удаление при удалении книги). `FileReferenceResolver` должен обрабатывать случай отсутствия файла обложки.
- Risks: Рассинхронизация между `Book.coverPath` в SwiftData и реальным наличием файла на диске при сбое во время импорта. Необходима атомарная операция: сначала сохранить файл, затем записать путь в SwiftData.

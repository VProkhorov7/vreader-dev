# ADR-006: Трёхуровневая модель состояния контента книги: cloudOnly / previewed / downloaded

**Status:** accepted
**Date:** 2026-03-29

## Context
Приложение работает с книгами из множества облачных источников. Пользователи хотят видеть свою библиотеку целиком, не скачивая все книги. При этом необходима возможность «полистать» книгу перед полной загрузкой, а также читать скачанные книги оффлайн. Нужна модель, которая скрывает от пользователя физическое расположение файла.

## Decision
Введена трёхуровневая модель `BookContentState`: `.cloudOnly` (только метаданные), `.previewed` (метаданные + первые 10 страниц в `Documents/Previews/{bookID}/`), `.downloaded` (полный файл в `Documents/Books/{bookID}/`). Превью скачиваются фоново при появлении книги в каталоге провайдера. При переходе `.previewed` → `.downloaded` превью удаляются автоматически. Нажатие «Читать» на `.cloudOnly` книге инициирует тихую фоновую загрузку с прогрессом на карточке.

## Alternatives
1. **Двухуровневая модель (cloudOnly / downloaded):** Простота реализации — отклонена, так как пользователь не может «полистать» книгу без полной загрузки, что критично для больших файлов (DJVU, PDF) на медленном соединении.
2. **Полная загрузка всех книг при подключении облака:** Максимальная доступность оффлайн — отклонена, так как библиотека может содержать тысячи книг, что приведёт к неприемлемому расходу хранилища и трафика.
3. **Потоковое чтение без локального кэша:** Минимальное использование хранилища — отклонено, так как делает невозможным оффлайн-чтение и значительно увеличивает latency при каждом перелистывании страницы.

## Consequences
- Positive: Пользователь видит полную библиотеку без скачивания всего контента. Возможность «полистать» книгу через превью улучшает UX. Экономия хранилища устройства. Плавный переход к оффлайн-чтению.
- Negative: Усложняется логика `DownloadManager` и `FileReferenceResolver`. Необходимо управлять тремя путями хранения (`Previews/`, `Books/`, `Covers/`). LRU-очистка при превышении порога (2GB) требует аккуратной реализации.
- Risks: Рассинхронизация состояния между `Book.contentState` в SwiftData и реальным наличием файла на диске. `FileReferenceResolver.repair()` должен корректно обрабатывать все три состояния. Фоновая загрузка превью может конкурировать с интерактивными загрузками за пропускную способность.

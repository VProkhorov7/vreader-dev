# ADR-009: Использование композиции @Observable классов вместо единого AppState

**Status:** accepted
**Date:** 2026-03-29

## Context
Приложение имеет сложное UI-состояние: навигация, библиотека, аудиоплеер, ридер — каждый со своим независимым жизненным циклом. Необходимо выбрать архитектуру управления состоянием, которая масштабируется с ростом функциональности, минимизирует лишние перерисовки SwiftUI и соответствует Swift 6 Concurrency.

## Decision
Состояние разбито на четыре независимых `@Observable` класса: `NavigationState` (вкладки, navigation path), `LibraryState` (текущая книга, поиск, фильтры), `PlayerState` (аудиокнига, позиция, воспроизведение), `ReaderState` (позиция, глава, настройки отображения). Все передаются через `@Environment`. God Object (`AppState`) явно запрещён архитектурными инвариантами.

## Alternatives
1. **Единый AppState (God Object):** Простота доступа из любого места — отклонён, так как любое изменение в одной части состояния вызывает перерисовку всех подписчиков, что критично для производительности ридера. Нарушает принцип единственной ответственности.
2. **Redux/TCA (The Composable Architecture):** Предсказуемый однонаправленный поток данных — отклонён, так как TCA является сторонней зависимостью, значительно увеличивает бойлерплейт, и его преимущества избыточны для приложения без сложной бизнес-логики на уровне UI.
3. **Combine + ObservableObject:** Реактивный подход с Combine — отклонён, так как `@Observable` (Swift 5.9+) предоставляет более гранулярное отслеживание зависимостей без необходимости явных `@Published` на каждом свойстве, что снижает количество лишних перерисовок.

## Consequences
- Positive: Гранулярные перерисовки SwiftUI — изменение `PlayerState` не перерисовывает `LibraryView`. Чёткое разделение ответственности. Лёгкое тестирование каждого класса состояния независимо. Соответствие Swift 6 Concurrency через `@MainActor`.
- Negative: Необходимо явно передавать нужные объекты состояния через `@Environment` в каждый view. При глубокой иерархии view может потребоваться передача нескольких объектов состояния.
- Risks: При неправильном разграничении ответственности между классами состояния возможно дублирование данных (например, `currentBook` в `LibraryState` и `ReaderState`). Требует чёткого соглашения о том, какой класс является источником истины для каждого куска данных.
